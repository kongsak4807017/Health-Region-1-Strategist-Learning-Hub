# Module 2.2: Fishbone Diagram (Ishikawa) - คู่มือการใช้งาน
## เอกสารประกอบการสอน

---

## 1. ความเป็นมา

Fishbone Diagram หรือ Ishikawa Diagram หรือ Cause-and-Effect Diagram เป็นเครื่องมือที่พัฒนาโดย Kaoru Ishikawa ในปี 1968 เพื่อวิเคราะห์สาเหตุของปัญหาคุณภาพในอุตสาหกรรม รูปร่างคล้ายกระดูกปลา (fishbone) จึงเป็นที่มาของชื่อ

---

## 2. โครงสร้าง Fishbone Diagram

### 2.1 ส่วนประกอบหลัก

```
                            [EFFECT]
                      (ปัญหาที่เกิดขึ้น)
                               |
       +-----------------------+-----------------------+
       |                       |                       |
  _____|_____             ______|______             ____|____
 /           \           /             \           /         \
PEOPLE    PROCESS      POLICY         PLACE       PRICE/TECH
```

### 2.2 หมวดหมู่สาเหตุ (Categories)

สำหรับสาธารณสุข ใช้ **4Ps หรือ 4Ms**:

| Category | ไทย | รายละเอียด |
|----------|-----|-----------|
| **People** | คน | ประชากร ผู้ให้บริการ ผู้กำหนดนโยบาย |
| **Process** | กระบวนการ | กระบวนการบริการ การส่งต่อ การจัดการข้อมูล |
| **Policy** | นโยบาย | นโยบายระดับชาติ ระเบียบ การจัดสรรทรัพยากร |
| **Place** | สถานที่/บริบท | โครงสร้างพื้นฐาน ภูมิศาสตร์ สภาพแวดล้อมทางสังคม |

หรือใช้ **6Ms** สำหรับกรณีทางคลินิก:
- Man (คน)
- Method (วิธีการ)
- Machine (เครื่องมือ)
- Material (วัสดุ)
- Measurement (การวัด)
- Mother Nature (สภาพแวดล้อม)

---

## 3. ขั้นตอนการสร้าง Fishbone Diagram

### Step 1: กำหนดปัญหา (Effect)
เขียนปัญหาที่หัวลูกศรด้านขวา ให้ชัดเจน เฉพาะเจาะจง

**ตัวอย่าง:**
- ✅ "อัตราตายมารดา 80/100,000"
- ❌ "สุขภาพแย่"

### Step 2: วาดกระดูกหลัก (Main Spine)
วาดเส้นแนวนอนจากซ้ายไปขวา ไปยังปัญหา

### Step 3: ระบุหมวดหมู่ (Categories)
วาดเส้นแยก (ribs) สำหรับแต่ละหมวดหมู่ 4Ps

### Step 4: Brainstorm สาเหตุ
เติมสาเหตุในแต่ละหมวดหมู่ ถาม "ทำไม?"

### Step 5: ลงลึกในแต่ละสาเหตุ
แตกย่อยสาเหตุหลักเป็นสาเหตุย่อย (sub-causes)

### Step 6: ระบุสาเหตุหลัก
วงกลมหรือเน้นสาเหตุที่สำคัญที่สุด

## 4. ตัวอย่าง Fishbone Diagram

### 4.1 ตัวอย่างที่ 1: อัตราตายมารดาสูง

```
                            [อัตราตายมารดาสูง
                              80/100,000]
                                     |
    +----------------+----------------+----------------+
    |                |                |                |
 ___|___         ____|____        ____|____        ___|___
/       \       /         \      /         \      /       \
PEOPLE  PROCESS  POLICY      PLACE

PEOPLE (คน):
├── มารดา
│   ├── อายุน้อย/มาก
│   ├── ไม่รู้จักสัญญาณอันตราย
│   └── ขาดการศึกษา
├── บุคลากร
│   ├── แพทย์เชี่ยวชาญไม่พอ
│   └── พยาบาลขาดทักษะ
└── ญาติ
    └── ไม่เข้าใจความสำคัญ

PROCESS (กระบวนการ):
├── ANC
│   ├── คัดกรองไม่เข้มงวด
│   └── ติดตามไม่ต่อเนื่อง
├── Referral
│   ├── ระบบส่งต่อล่าช้า
│   └── ไม่มีทีมสูติอุบัติเหตุ
└── Emergency
    └── ไม่มี protocol ชัดเจน

POLICY (นโยบาย):
├── การเงิน
│   ├── ไม่มีค่าเดินทาง
│   └── ไม่ครอบคลุมค่ารักษา
├── บุคลากร
│   └── ไม่มี incentive ทำงานพื้นที่ห่างไกล
└── มาตรฐาน
    └── ไม่มีระบบ early warning

PLACE (สถานที่):
├── ภูมิศาสตร์
│   ├── ห่างไกลจาก รพ.
│   └── ถนนไม่ดี
├── โครงสร้าง
│   ├── รพ.สต. ไม่มีอุปกรณ์
│   └── รพ.ไม่มี blood bank
└── สังคม
    ├── ความเชื่อเรื่องการคลอด
    └── การตัดสินใจช้า
```

### 4.2 ตัวอย่างที่ 2: โรคเบาหวานไม่ควบคุม

```
                      [HbA1c > 9%
                       ใน 40% ผู้ป่วย]
                              |
    +-------------+-------------+-------------+
    |             |             |             |
  __|__        ___|___       ___|___       ___|__
 /     \      /       \     /       \     /     \
People Process  Policy    Place

People:
├── ผู้ป่วย
│   ├── ไม่เข้าใจโรค
│   ├── ไม่ทานยาตามใบสั่ง
│   ├── กินอาหารไม่ควบคุม
│   └── ไม่ออกกำลังกาย
└── ผู้ดูแล
    └── ไม่มีความรู้ดูแลผู้ป่วย

Process:
├── วินิจฉัย
│   └── ตรวจไม่สม่ำเสมอ
├── รักษา
│   ├── ไม่ปรับยาตามผล
│   └── ไม่ส่งต่อผู้ป่วยซับซ้อน
└── Follow-up
    ├── ไม่มีระบบนัดหมาย
    └── ไม่ติดตามผู้ป่วยขาดนัด

Policy:
├── ยา
│   └── บัญชีจำกัด
├── เครื่องมือ
│   └── HbA1c ไม่ครอบคลุม
└── บุคลากร
    └── ผู้ป่วยต่อแพทย์มาก

Place:
├── รพ.สต.
│   └── ไม่มีเครื่องตรวจน้ำตาล
├── ชุมชน
│   └── อาหารขายแต่ของหวาน
└── บ้าน
    └── ไม่มีตาชั่ง ที่วัดน้ำตาล
```

## 5. เทคนิคการใช้ Fishbone อย่างมีประสิทธิภาพ

### 5.1 DO's (ควรทำ)

| เทคนิค | คำอธิบาย |
|--------|---------|
| ทำเป็นทีม | มีหลายมุมมอง |
| Brainstorm ก่อน | คิดให้ครบ อย่า filter เร็ว |
| ถาม "Why" ซ้ำ | แตกย่อยจนถึง root cause |
| ใช้ Post-it | ง่ายต่อการจัดกลุ่ม |
| Focus ที่ facts | ไม่ใช่ความคิดเห็น |
| ตรวจสอบความสัมพันธ์ | ดูว่าสาเหตุเชื่อมโยงกันอย่างไร |

### 5.2 DON'Ts (ไม่ควรทำ)

| ข้อห้าม | ตัวอย่าง |
|---------|---------|
| สาเหตุซ้ำซ้อน | ใส่สาเหตุเดียวกันหลายหมวด |
| สาเหตุกว้างเกิน | "รัฐบาลไม่ดี" |
| โทษบุคคล | "เพราะพยาบาล" |
| หยุดเร็วเกินไป | ไม่แตกย่อยสาเหตุ |
| ไม่มี priority | ไม่ระบุว่าอันไหนสำคัญกว่า |

## 6. การเลือกสาเหตุหลักจาก Fishbone

### 6.1 เกณฑ์การเลือก

**หลังจากทำ Fishbone เสร็จ ให้ถาม:**

1. **Impact:** สาเหตุนี้มีผลต่อปัญหามากแค่ไหน?
2. **Controllability:** เราควบคุมสาเหตุนี้ได้หรือไม่?
3. **Frequency:** สาเหตุนี้เกิดบ่อยแค่ไหน?
4. **Evidence:** มีข้อมูลสนับสนุนหรือไม่?

### 6.2 การให้คะแนน

ใช้ตารางประเมิน:

| สาเหตุ | Impact (1-5) | Control (1-5) | Evidence (1-5) | รวม |
|--------|-------------|---------------|----------------|-----|
| ไม่มีแพทย์ | 5 | 3 | 5 | 13 |
| ถนนไม่ดี | 3 | 2 | 4 | 9 |
| คนไม่รู้ | 4 | 4 | 3 | 11 |

## 7. การใช้ Fishbone ร่วมกับเครื่องมืออื่น

### 7.1 Fishbone + 5 Whys

```
Fishbone: ระบุสาเหตุทั้งหมด
    ↓
เลือกสาเหตุสำคัญ 2-3 ข้อ
    ↓
ใช้ 5 Whys ลงลึกในแต่ละสาเหตุ
    ↓
ได้ Root Causes ที่ชัดเจน
```

### 7.2 Fishbone + Pareto

```
Fishbone: ระบุสาเหตุทั้งหมด
    ↓
เก็บข้อมูลความถี่ของแต่ละสาเหตุ
    ↓
ใช้ Pareto Analysis (80/20)
    ↓
Focus ที่สาเหตุ 20% ที่สร้างปัญหา 80%
```
